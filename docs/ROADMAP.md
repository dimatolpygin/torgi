# Дорожная карта реализации — bron-bot

Бот автоматической брони торгового места на `gorod.it-minsk.by` (Минский Комаровский
рынок, овощи). Подаёт заявку в 00:00 по Минску быстрее конкурентов, проверяет результат
по списку броней в ЛК и шлёт уведомление в Telegram.

11 этапов от пустого репозитория до боевой готовности. Каждый этап заканчивается
**демонстрируемым результатом**.

> **Где смотреть прогресс**: текущий статус всех этапов — в [`STATUS.md`](STATUS.md).
> Закрытие этапа сопровождается git-тегом `stage-N-done`.

> **Важное ограничение тест-периода**: пока брони на сайте недоступны (открываются ночью
> в 00:00, разбираются за минуты), реальную успешную подачу проверить нельзя. Поэтому этапы
> 2–4 верифицируются на «пустом» состоянии (бот корректно видит «дат нет» и формирует
> запрос), а финальная боевая проверка реальной подачи — отдельный пост-этап на белорусском
> сервере в боевую ночь.

## Этап 0 — Каркас репозитория

**Цель**: пустой, но рабочий проект со всей инфраструктурой для следующих этапов.

**Что делаем**:
- `package.json`, зависимости (undici, telegraf, pino, pg, ioredis, luxon, dotenv).
- `docker-compose.yml`: сервисы app, postgres, redis.
- `Dockerfile`, `.env.example`, `.gitignore`.
- Базовый `src/index.js`: подключается к Postgres и Redis, пишет лог через pino, держится живым.
- Логгер по правилам проекта (pino + pino-pretty, русские сообщения).

**Критерии приёмки** (закрытие → tag `stage-0-done`):
- [x] `docker compose up` поднимает три контейнера без ошибок
- [x] В логе видно успешное подключение к Postgres и Redis
- [x] Лог на русском, читаемый, с датой/временем
- [x] `.env.example` содержит все нужные переменные, секретов в репозитории нет

## Этап 1 — Авторизация на сайте

**Цель**: бот умеет войти в кабинет и подтвердить, что вход успешен.

**Что делаем**:
- Модуль `auth`: POST на `/rinki/minsk/login/` с `n_login`/`n_pass`.
- Сохранение сессионной куки `gorodid`.
- Проверка входа: парсинг ЛК на «пользователя <логин>» / ФИО.

**Критерии приёмки** (закрытие → tag `stage-1-done`):
- [x] Бот логинится под тестовым аккаунтом, в логе «вошли как <ФИО>»
- [x] При неверном пароле бот понимает, что вход не удался, и пишет это в лог
- [x] Сессионная кука сохраняется и переиспользуется между запросами

---

## Этап 2 — Чтение состояния рынка

**Цель**: бот видит реальное состояние формы по Комаровскому рынку.

**Что делаем**:
- Выбор рынка Комаровский (`rinok=10`).
- Вызовы AJAX `type_mest/`, `limit_mest/`, `calend/`.
- Парсинг: есть ли доступная дата, какой тип места по умолчанию.

**Критерии приёмки** (закрытие → tag `stage-2-done`):
- [x] Бот корректно сообщает «дат нет» в дневное время (пустое состояние)
- [x] Бот извлекает тип места по умолчанию и список дат из ответов AJAX
- [x] Ответы сервера логируются для разбора боевого формата

---

## Этап 3 — Формирование заявки (dry-run)

**Цель**: бот собирает точный запрос на подачу, но пока не отправляет вслепую.

**Что делаем**:
- Сборка POST на `/rinki/minsk/reg/fiz/`: `reg=1`, `rinok=10`, `type_mesta`, `arr_date`,
  `assort_arr[]=2` (овощи), служебные поля.
- Режим dry-run: запрос формируется и логируется, реальная отправка под флагом.

**Критерии приёмки** (закрытие → tag `stage-3-done`):
- [x] В логе виден полностью сформированный запрос, готовый к отправке
- [x] Ассортимент жёстко = овощи, рынок = Комаровский, дата берётся из состояния
- [x] Флаг `DRY_RUN` управляет реальной отправкой

**Доп. проверка (диагностика сервера):** запрос с пустой датой → `code 500, {arr_date:"no_date"}`,
поля персоны не помечены ошибкой → для залогиненного нужна только дата + ассортимент.

## Этап 4 — Верификация через ЛК

**Цель**: бот подтверждает успех не по ответу формы, а по факту в списке броней.

**Что делаем**:
- Парсинг таблицы «Для Вас забронированы следующие места» в `account/`.
- Сопоставление: появилась ли запись на нужную дату, Комаровский, овощи.

**Критерии приёмки** (закрытие → tag `stage-4-done`):
- [x] Бот выводит актуальный список броней аккаунта из ЛК
- [x] Бот умеет ответить «есть ли бронь на дату X» по содержимому ЛК

---

## Этап 5 — Точный планировщик и тайминг

**Цель**: бот сам просыпается к полуночи и стреляет с высокой точностью.

**Что делаем**:
- Пробуждение заранее (≈23:58), прогрев сессии и keep-alive соединения.
- Точный выстрел в 00:00:00.000 по `Europe/Minsk` через hrtime-ожидание.
- Пропуск ночей на понедельник (регистрации нет).
- Режим «долбёжки» при отсутствии даты (редкие сбои: места открываются позже).

**Критерии приёмки** (закрытие → tag `stage-5-done`):
- [x] При подставной целевой минуте бот срабатывает в заданную секунду, отклонение в логе ≤ несколько мс
- [x] Ночь на понедельник пропускается
- [x] Если даты нет в 00:00 — бот переходит в режим повторных проверок, а не сдаётся

## Этап 7 — Telegram-уведомления

**Цель**: результат каждой ночи приходит в Telegram.

**Что делаем**:
- Бот на telegraf, регистрация chat_id через `/start`.
- Сообщения: «подано», «подтверждено (место, дата)», «ошибка — подайте вручную».

**Критерии приёмки** (закрытие → tag `stage-7-done`):
- [x] После прогона в Telegram приходит сообщение с результатом по каждому аккаунту
- [x] При неуспехе приходит явное предупреждение подать вручную
- [x] Команда `/status` отдаёт состояние сервера (аптайм, след. подача, режим)
- [x] Алерт о возможной блокировке IP приходит при серии ошибок на подаче

## Этап 8 — Хранилище и история

**Цель**: каждая попытка фиксируется, состояние переживает перезапуск.

**Что делаем**:
- Схема Postgres: аккаунты, попытки, результаты, отклонение тайминга.
- Redis: хранение актуальных сессионных кук с TTL, флаги «сегодня подано».

**Критерии приёмки** (закрытие → tag `stage-8-done`):
- [ ] После прогона в Postgres появляется запись о попытке с деталями и результатом
- [ ] Сессионная кука берётся из Redis при повторном запуске, лишнего логина нет

## Пост-этап (вне тест-периода) — Боевая верификация

Не входит в тест-фазу: реальная успешная подача в 00:00 на белорусском сервере, когда
брони открыты. Проверяется в режиме подстраховки (бот + ручная подача клиента
параллельно), несколько ночей подряд, до перехода в автономный режим.
